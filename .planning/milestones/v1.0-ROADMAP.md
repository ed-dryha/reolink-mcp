# Roadmap: reolink-mcp

## Overview

Four vertical slices from empty repo to published PyPI package. Phase 1 delivers the core value end-to-end: a user configures cameras in YAML, adds the server to a real MCP client, and gets a live downscaled snapshot from a real camera — with the session-limit and error-handling machinery (the project's biggest field risk) proven from day one. Phase 2 completes the observe surface and validates it live on both target cameras. Phase 3 adds the safe control surface (deterrence, zoom, preset PTZ) with capability gating, tool annotations, and read-only mode. Phase 4 packages, documents, and publishes to PyPI and the MCP registry with hardware-free CI.

## Phase Details

### Phase 1: Core Snapshot Slice

**Goal**: As a Reolink camera owner using an MCP client (Claude Code/Desktop), I want to configure my cameras in YAML with env-var-only passwords and get a live downscaled snapshot from a real camera by name, so that I can see my cameras directly through Claude with no NVR or home-automation daemon in between — proven session-safe against shared hardware.
**Mode:** mvp
**Depends on**: Nothing (first phase)
**Requirements**: CONN-01, CONN-02, CONN-03, CONN-04, CONN-05, OBSV-01, OBSV-04, SAFE-03, HDWR-03
**Success Criteria** (what must be TRUE):

  1. User can define cameras (name, host, username) in a YAML file with passwords supplied only via `RMCP_CAMERAS__<camera_name>__PASSWORD`-style env vars (named-map config shape, corrected from the `__0__` list-index example during Phase 1 research); a password in YAML or a missing env var fails loudly at startup with a clear message
  2. From a real MCP client (Claude Code or Claude Desktop), user can run `list_cameras` and see all configured cameras with connection status
  3. User asks for a snapshot and receives a live image from a real camera as a native MCP image content block, unconditionally downscaled server-side (~1280px) so it never blows client token/size limits
  4. Offline camera, wrong credentials, and session-limit (`rspCode: -5`) conditions each produce a distinct, actionable error message naming the camera and likely cause — never a raw traceback
  5. Server runs alongside surveillance-security-ai holding sessions on the same cameras without session exhaustion, and repeated restarts leak no session slots (lazy login, guaranteed logout on shutdown); all logging goes to stderr with stdout reserved for the MCP protocol

**Plans**: 4 plans (4 waves — sequential: config → manager/errors → server+list_cameras → get_snapshot+hardware validation)

### Phase 2: Full Observe Surface

**Goal**: Users can fully inspect their cameras — hardware details, capabilities, live device state, and current AI detection flags — validated end-to-end on the real P437 and P320
**Mode:** mvp
**Depends on**: Phase 1
**Requirements**: OBSV-02, OBSV-03, OBSV-05, OBSV-06, HDWR-01
**Success Criteria** (what must be TRUE):

  1. User can get model, firmware, and hardware details for any configured camera via `get_device_info`
  2. User can see what a camera supports (zoom, spotlight, siren, PTZ, ...) via `get_capabilities`, and this capability data is available to gate other tools' behavior
  3. User can read current device state (day/night mode, lights, siren, detection flags) via `get_states`
  4. User can check current person/vehicle/animal AI detection via `get_recent_events`, with the tool description explicit that this is current state from on-demand polling, not history
  5. The full observe surface (snapshot, states, AI detection flags) is validated end-to-end against the real P437 and P320 from an MCP client

### Phase 3: Safe Controls & Safety Rails

**Goal**: Users can operate deterrence (siren, spotlight, IR, white LED), optical zoom, and preset-based PTZ — every control capability-gated and annotated so clients can gate approvals, with a read-only startup mode
**Mode:** mvp
**Depends on**: Phase 2
**Requirements**: CTRL-01, CTRL-02, CTRL-03, CTRL-04, CTRL-05, CTRL-06, CTRL-07, CTRL-08, CTRL-09, CTRL-10, SAFE-01, SAFE-02, HDWR-02
**Success Criteria** (what must be TRUE):

  1. User can trigger the siren with an auto-off duration, toggle the spotlight, control IR lights and the white LED, and drive optical zoom — validated live on the P437 (siren, spotlight, zoom) and P320 (IR)
  2. User can enumerate PTZ presets, move to a named preset, read the current PTZ position, and configure/trigger the guard position (mock-validated; PTZ hardware pending)
  3. Invoking a control the camera doesn't support refuses cleanly with a capability-based message ("this camera has no siren") instead of a raw API error
  4. Every registered tool carries correct MCP annotations (`readOnlyHint: true` on observe tools, `destructiveHint` on physical-effect controls like the siren), verified by an annotation-completeness unit test over the full tool registry
  5. Starting the server with `RMCP_READ_ONLY=true` strips all control tools — an MCP client sees only observe tools

**Plans**: 3 plans (3 waves — sequential: deterrence+read-only foundation -> zoom+PTZ -> ptz_guard+safety completion+live QA)

### Phase 4: Packaging & Release

**Goal**: Anyone with a Reolink camera can install and run the server with one line and copy-paste client config — published, listed, CI-protected, and documented
**Mode:** mvp
**Depends on**: Phase 3
**Requirements**: REL-01, REL-02, REL-03, REL-04
**Success Criteria** (what must be TRUE):

  1. User can run `uvx reolink-mcp` from a clean environment and get a working server, installed from PyPI
  2. Server is listed in the MCP registry and discoverable by name
  3. CI (GitHub Actions) runs ruff + pytest with mocked `reolink-aio` on every PR — no camera hardware required — including stdout-purity enforcement (ruff T201)
  4. README quickstart lets a new user go from zero to a working Claude Desktop/Code setup via copy-paste config JSON, with camera compatibility notes and the concurrent-session caveat documented prominently

**Plans**: 4 plans (3 waves -- wave 1: CI protection + README/docs in parallel; wave 2: release engineering artifacts; wave 3: human-guided release execution)
