# Requirements: reolink-mcp

**Defined:** 2026-07-08
**Core Value:** A user with a Reolink camera adds `reolink-mcp` to their MCP client config, asks "show me the front door camera," and gets a live snapshot — direct to camera, no NVR or home-automation daemon in between.

**v1 target hardware:** Reolink P437 (4K 8MP, optical zoom, spotlight, siren, AI detection) and P320 (5MP fixed bullet, IR, AI detection) — both owned by the author and available for live validation. v1 covers the full feature surface of these two cameras; PTZ preset tools ship mock-validated (PTZ hardware planned). Future versions extend to capabilities of additional cameras/sensors as acquired.

## v1 Requirements

Requirements for initial release. Each maps to roadmap phases.

### Connection & Config

- [x] **CONN-01**: User can configure one or more cameras in a YAML file (name, host, username)
- [x] **CONN-02**: User can supply camera passwords via environment variables (`RMCP_CAMERAS__<camera_name>__PASSWORD`-style nested overrides — named map, not list index; see Phase 1 research); passwords are never read from YAML
- [x] **CONN-03**: Server maintains one long-lived `reolink-aio` session per camera — lazy login on first use, guaranteed logout on shutdown (no session-slot leaks)
- [x] **CONN-04**: User gets a distinct, actionable error message when a camera's concurrent-session limit is hit (`rspCode: -5`), naming the caveat and likely cause
- [x] **CONN-05**: User gets actionable error text for offline cameras and wrong credentials (e.g., "camera 'garage' unreachable at 192.168.1.44 — check power/network"), never raw tracebacks

### Observe Tools

- [x] **OBSV-01**: User can list all configured cameras with connection status via `list_cameras`
- [x] **OBSV-02**: User can get model, firmware, and hardware details via `get_device_info` (one consolidated tool)
- [x] **OBSV-03**: User can see what a camera supports (zoom, spotlight, siren, PTZ, …) via `get_capabilities`; capability data gates all other tools' behavior
- [x] **OBSV-04**: User can get a live snapshot as a native MCP image content block via `get_snapshot`, unconditionally downscaled server-side (~1280px default) to stay within client token/size limits
- [x] **OBSV-05**: User can read current device state (day/night mode, lights, siren, detection flags) via `get_states`
- [x] **OBSV-06**: User can check current AI detection states (person/vehicle/animal) via `get_recent_events` — on-demand polling; tool description is explicit that this is current state, not history

### Control Tools

- [x] **CTRL-01**: User can trigger the siren with an auto-off duration via `set_siren`
- [x] **CTRL-02**: User can toggle the spotlight via `set_spotlight`
- [x] **CTRL-03**: User can control IR lights via `set_ir_lights`
- [x] **CTRL-04**: User can control the white LED via `set_white_led`
- [x] **CTRL-05**: User can control optical zoom via `set_zoom` (validated on P437)
- [x] **CTRL-06**: User can enumerate camera-defined PTZ presets via `list_presets`
- [x] **CTRL-07**: User can move a PTZ camera to a named preset via `ptz_move_to_preset`
- [x] **CTRL-08**: User can read the current PTZ position via `ptz_position`
- [x] **CTRL-09**: User can configure/trigger the PTZ guard position via `ptz_guard`
- [x] **CTRL-10**: Every control tool checks camera capabilities first and refuses cleanly ("this camera has no siren") instead of surfacing raw API errors

### Safety

- [x] **SAFE-01**: Every tool carries MCP tool annotations — `readOnlyHint: true` on observe tools, `destructiveHint` on physical-effect controls (siren) — so clients can gate approvals
- [x] **SAFE-02**: User can start the server in read-only mode (`RMCP_READ_ONLY=true`) which strips all control tools at startup
- [x] **SAFE-03**: All logging goes to stderr only; stdout is reserved for the MCP protocol (enforced by lint rule and a stdout-purity test)

### Hardware Validation

- [x] **HDWR-01**: Full observe surface validated end-to-end against the real P437 and P320 from an MCP client (snapshot, states, AI detection flags on both)
- [x] **HDWR-02**: Deterrence and zoom controls validated on the P437 (siren, spotlight, zoom) and IR on the P320
- [x] **HDWR-03**: Server coexists with surveillance-security-ai holding sessions on the same cameras without session exhaustion during normal operation

### Release

- [x] **REL-01**: Package published to PyPI and runnable via `uvx reolink-mcp`
- [x] **REL-02**: Server listed in the MCP registry
- [x] **REL-03**: CI (GitHub Actions) runs ruff + pytest with mocked `reolink-aio` on every PR — no camera hardware required; includes stdout-purity enforcement (ruff `T201`)
- [x] **REL-04**: README quickstart with copy-paste Claude Desktop/Code config JSON, camera compatibility notes, and the concurrent-session caveat documented prominently

## v2 Requirements

Deferred to future release. Tracked but not in current roadmap.

### Connectivity & Scale

- **CONN-V2-01**: NVR / Home Hub channel support — one host entry fans out to N cameras (needed for battery cams; cheap via reolink-aio `Host`)
- **CONN-V2-02**: Streamable HTTP transport + bearer auth for remote/Docker deployment
- **CONN-V2-03**: LAN camera auto-discovery

### Events & Media

- **EVNT-V2-01**: Event push subscriptions (ONVIF SWN / Baichuan TCP) exposed as MCP resources/notifications — headline post-v1 feature
- **EVNT-V2-02**: VOD/recording search with save-to-disk clips (path returned as text)

### Tools & Ergonomics

- **TOOL-V2-01**: `max_width` parameter on `get_snapshot` for multi-camera sweep ergonomics
- **TOOL-V2-02**: Low-risk setting controls (day/night mode, detection sensitivity) with `destructiveHint`
- **TOOL-V2-03**: Consolidated multi-camera state summary tool
- **TOOL-V2-04**: Wider control surface as hardware arrives: auto-tracking, two-way audio talk-down, quick replies
- **TOOL-V2-05**: Support capabilities of next cameras/sensors as acquired (per-device capability extension is the growth pattern)

## Out of Scope

Explicitly excluded. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Live video streaming through MCP | MCP has no video content type; base64 video destroys context; return snapshots + RTSP URL as text instead |
| Risky settings surface (privacy mask, encoding, network, recording config — ~59 reolink-aio setters) | Mistakes are user-visible and hard to reverse; LLM misfires become real-world incidents |
| Runtime `connect_camera(host, user, password)` tool | Credentials would flow through LLM chat history and client logs (hanwha-mcp anti-pattern) |
| Reboot / firmware / factory tools | Destructive; kills sessions shared with other systems (author's own constraint) |
| Continuous/absolute PTZ movement | Open-loop movement from an LLM with no live video feedback; preset-based only |
| One micro-tool per state field | Tool-count bloat degrades LLM tool selection (dahua-mcp anti-pattern); consolidated tools instead |
| Any Frigate / Home Assistant / surveillance-security-ai dependency | Direct-to-camera standalone is the product's reason to exist |
